1. Ideally, we want to use models from a similar family that shares a tokenizer. If given two models with differing ones, one option would be to align each token candidate in the text space instead of a vector space. For example, we could first grab the next-token distribution from the expert model to get V_head. Then, for each expert candidate token in V_head, we can decode that to a small piece of text, and re-encode with the amateur tokenizer. This way, we are describing the same text space, so the probability distributions are comparable. We can then apply contrastive scoring on this vector space. However, this method is extremely computationally expensive, since it needs to be done on every candidate token, and ideally, we would simply use models of the same family.

2. It seems like contrastive decoding would not be used in practice, since it requires prompting two models. This is computationally expensive and in production teams would likely get better results by doing small tweaks to a single model or use one model for re-generation and another for some ranking algorithm. I imagine that it would have more practical use in a research environment, since the idea of essentially "tuning" the output of a model using this contrastive difference would give researchers a lot more control over the LM and can help them detect repetition and reduce incoherent text.
